Kite RTOS

An educational real-time kernel for ARM Cortex‑M

32 priority levels · Preemptive · MPU stack guard · Designed for clarity

1. Philosophy & architecture

Kite RTOS is a compact, preemptive kernel written for Cortex‑M MCUs (STM32, etc.). It provides tick-driven scheduling, blocking synchronization (semaphores, mutexes with priority inheritance), and a clean SVC‑based system call interface. The design emphasizes deterministic behaviour, low overhead, and educational transparency.

Core design pillars:
• Round‑robin time slicing (configurable ticks).
• Bitmap ready-queue (O(1) highest priority lookup).
• Sleep list sorted by wakeup time (delta‑list).
• Critical sections via BASEPRI masking.
• MPU stack guard for stack overflow detection.

2. Key data structures

TCB_t
Task Control Block: holds PSP, state+priority (packed), blocking counters, mutex ownership bitmap, waiting queues pointers. Embedded in doubly linked ready list and sleep/wait queues.
semaphore_t
Counting semaphore: count + priority‑sorted waiting queue (wq_head/wq_bitmap).
mutex_t
Mutex with priority inheritance: owner, highest waiter priority, wait queue, and nested holding per TCB.
Ready queue
Array of 32 lists, plus a 32‑bit bitmap. Task enqueue/dequeue O(1) per priority.

The TCB packs state_prio into a 16‑bit field: 2 bits state (SLEEP/WAKE/BLOCKED), 5 bits base priority, 5 bits effective priority (used for inheritance). Fast bit manipulations via macros TCB_EFF_PRIO() etc.

Every tick: increment global tick, traverse sleep list and move ready tasks. If time slice counter reaches SCHED_TIME_SLICE, context switch is requested via PendSV. PendSV is the lowest priority exception, ensuring clean context switch.

⚡ SVC calls: All kernel services (task_delay, sem_wait, mutex_lock, …) use SVC instructions. SVC_Handler extracts the immediate number and forwards to the right service, running with kernel privileges.

5. Synchronization: semaphores & mutexes

📌 Counting semaphores

semaphore_init(sem, initial_count) – semaphore_wait() decrements count, blocks if negative. semaphore_post() increments, unblocks highest‑priority waiter.

sequenceDiagram participant TaskLow participant TaskHigh participant Sem as Semaphore(count=0) TaskHigh->>Sem: sem_wait() -> blocks TaskLow->>Sem: sem_post() Sem->>TaskHigh: unblock (highest priority) TaskHigh->>TaskHigh: runs

🔒 Mutex with priority inheritance

mutex_init() – recursive ownership tracking: each TCB holds a bitmap of owned mutexes (max HELD_MUTEX_MAX). When a higher-priority task waits on a mutex, the owner's effective priority inherits the waiter's priority to avoid unbounded priority inversion.

mutex_t console_mutex; void task_uart(void) { mutex_lock(&console_mutex); // critical output mutex_unlock(&console_mutex); }
Priority inheritance logic: When a task blocks on a mutex, propagate_priority() walks the chain of mutex owners and boosts their effective priority. After unlock, the original base priority is restored if no longer needed.

6. Internal kernel flow – from SVC to reschedule

All system calls are marshalled via SVC. Example: task_delay(10) triggers svc 2 → SVC_Handler_C calls svc_task_delay. In critical section it sets block_count = global_systick+ticks, removes from ready queue and inserts into sleep list. Then PendSV is triggered.

flowchart LR A[User task] -->|SVC #2| B[SVC_Handler] B --> C[Extract SVC number] C --> D[svc_task_delay] D --> E[ENTER_CRITICAL] E --> F[Remove from ready queue & add to sleep list] F --> G[EXIT_CRITICAL] G --> H[request_context_switch
PendSV set] H --> I[PendSV_Handler runs reschedule]

Critical sections & interrupt masking

sched_enter_critical() raises BASEPRI to KERNEL_INTERRUPT_PRIORITY (level 15) — disables interrupts with priority ≥15. Nested critical sections are counted.

MPU stack overflow guard

Each task stack is followed by a 32‑byte guard region (size configurable via STACK_GUARD_SIZE_WORDS). The MPU region #1 is reconfigured on each context switch to protect the current task’s guard area → any overflow triggers MemManage fault.

8. Mutex lock with inheritance — visual walk

graph TD start[mutex_lock called] --> check{Locked?} check -->|No| own[Owner = current task, held slot set] own --> done[Lock acquired] check -->|Yes| block[Task state = BLOCKED, waiting_on = mutex] block --> add_wq[Add to mutex wait queue sorted by priority] add_wq --> prop[propagate_priority: boost owner's effective priority] prop --> pend[PendSV -> scheduler runs next task] pend --> unlock_event[Eventually mutex_unlock] unlock_event --> transfer[Highest waiter becomes new owner]

Priority propagation: while the mutex is locked, the owner runs at the priority of the highest waiting task (if higher), thus preventing unbounded inversion.

10. Under the hood: interesting routines

  • rq_add / rq_remove – circular doubly linked ready list per priority.
  • sl_insert – inserts task into sleep list sorted by absolute wake time (delta method).
  • wq_enqueue / wq_dequeue_highest – priority sorted wait queues for semaphores and mutexes.
  • held_alloc_slot / held_set – track nested mutexes per task (up to HELD_MUTEX_MAX).
  • propagate_priority – while walking mutex ownership chain, boost owner to priority of highest waiter.
  • Stack initialization: initial context frame guarantees that first time task runs, handler function is loaded with correct CPU registers (xPSR, PC, LR, R0-R3).
Pro tip: The kernel uses __builtin_clz for fast bitmap → highest priority ready task; global_systick is 32-bit and wraps, but delta-comparisons use signed arithmetic to keep correctness.

11. Boot sequence from reset to first task

sequenceDiagram participant Reset as Reset_Handler participant Main as main() participant Kite as kite_start() participant Sched as scheduler_init() participant SysTick as SysTick_Handler participant PendSV as PendSV_Handler Reset->>Main: C startup, SystemInit Main->>Kite: create tasks, then kite_start() Kite->>Kite: create_idle_task() Kite->>Sched: branch to scheduler_init Sched->>Sched: init_helper, set MSP, task_stack_init Sched->>Sched: scheduler_start(), enable SysTick, MPU guard Sched->>PendSV: svc 0 (start first task) PendSV->>PendSV: load PSP of highest ready task PendSV->>SysTick: periodic tick fires, yields when needed

The very first SVC call (svc_start_first_task) loads the PSP of the highest priority task, sets CONTROL register to use PSP, and returns to user mode — the task begins execution.

Learning takeaways: This RTOS demonstrates real-world techniques: bitmap scheduling, O(1) priority selection, MPU-assisted stack overflow detection, priority inheritance for mutexes, and minimal interrupt latency via BASEPRI critical sections. Each component works together to build a deterministic, preemptive kernel suitable for embedded control systems.
