01 — Introduction

What is Kite RTOS?

Imagine you're a chef in a busy restaurant. You have to cook three dishes at the same time — a soup, a steak, and a salad. You don't literally do all three simultaneously; you switch between them quickly, checking the steak while the soup simmers. A Real-Time Operating System (RTOS) does exactly this for a microcontroller — it lets multiple tasks share one CPU by switching between them so fast it feels simultaneous.

Kite RTOS is an educational, production-quality kernel written in C for ARM Cortex-M processors (like those on STM32 boards). It provides everything needed for deterministic, multi-task embedded software.

⚡

Preemptive Scheduling

Higher-priority tasks instantly interrupt lower-priority ones. No task can hog the CPU unfairly.

🗓️

Round-Robin Time Slicing

Equal-priority tasks share the CPU in configurable time slices. Every task gets a fair turn.

🔒

Priority Inheritance

Prevents priority inversion — a low-priority task temporarily borrows the priority of a high-priority waiter.

🛡️

MPU Stack Guard

Hardware memory protection catches stack overflows per-task before they corrupt other data.

🧩

SVC System Calls

Clean ARM SVC-based interface for all kernel services — delay, yield, semaphore, mutex.

🎯

O(1) Scheduler

32-bit bitmap lets the scheduler find the highest-priority ready task in a single CPU instruction.

04 — Scheduling

Scheduler & Priority Queues

Ready Queue — O(1) with a Bitmap

The scheduler maintains an array of 32 circular doubly-linked lists — one per priority level — plus a 32-bit ready_bitmap where bit N is set if any task at priority N is ready.

Finding the highest-priority ready task takes one CPU instruction:

rq_highest() — sched.c
static inline TCB_t *rq_highest(void)
{
    if (ready_bitmap == 0U) return NULL;

    /* __builtin_clz = Count Leading Zeros
       31 - clz(bitmap) = index of highest set bit
       e.g. bitmap = 0b00101000 → clz=2 → p=5         */
    uint8_t p = (uint8_t)(31U - __builtin_clz(ready_bitmap));
    return ready_queue[p];
}

Round-Robin within a Priority Level

Tasks of the same priority share the CPU using a circular doubly-linked list. After each time slice, the "head" pointer advances to the next task:

Figure 5 — Circular doubly-linked list for round-robin at priority 5

Sleep List — Sorted by Wake Time

When a task calls task_delay(N), it's removed from the ready queue and inserted into a singly-linked list sorted by its absolute wake time (global_systick + N). Every SysTick, the list head is checked — if global_systick >= block_count, the task wakes up.

⚠️
Tick Wrap-around Handled Correctly

global_systick is a 32-bit counter that wraps after ~49 days at 1kHz. Comparisons use signed arithmetic: (int32_t)(global_systick - block_count) >= 0 — this handles wrap correctly without special cases.

Stack Allocation Logic

find_stack_area() — sched.c
static uint32_t *find_stack_area(
    uint32_t stack_words,
    uint32_t **guard_base_out)
{
    /* Round up to 8-word boundary */
    stack_words = (stack_words + 7U) & ~7U;

    new_task_psp  = next_task_psp;
    next_task_psp -= stack_words;        /* carve stack */

    /* Guard page: 32-byte aligned just below stack */
    uint32_t guard_addr =
        ((uint32_t)next_task_psp) & ~31U;

    *guard_base_out = (uint32_t *)guard_addr;
    return new_task_psp;
}

Initial Stack Frame Setup

Before the scheduler runs, task_stack_init() pre-populates each task's stack so the first context restore works correctly:

task_stack_init() — initial frame
*(--stack) = 0x01000000U;  /* xPSR: Thumb bit */
*(--stack) = (uint32_t)handler; /* PC: task entry  */
*(--stack) = (uint32_t)kernel_panic; /* LR: if task returns → panic */
*(--stack) = 0x0000000C;  /* R12 */
*(--stack) = 0x00000003;  /* R3  */
*(--stack) = 0x00000002;  /* R2  */
*(--stack) = 0x00000001;  /* R1  */
*(--stack) = 0x00000000;  /* R0  */
/* + 8 words of zeros for R4-R11 */
✅
Why kernel_panic as the LR?

Tasks in an RTOS should never return — they run infinite loops. If one accidentally returns, the LR (link register) points to kernel_panic() which spins forever, making the bug visible.

Why does scheduler_init reset the MSP?

Before calling init_helper(), the kernel records the current MSP position (after all task stacks are carved). It then resets MSP to this saved value — this reclaims the stack space used for task stack pointers as clean kernel stack. This way the kernel's own stack stays above all task stacks without overlap.

08 — Timer & Tick

SysTick Handler — The Kernel Heartbeat

Every millisecond (at TICK_HZ = 1000), SysTick fires. This is the heartbeat of the entire kernel:

SysTick_Handler — sched.c
void SysTick_Handler(void)
{
    global_systick++;   /* advance the master clock */
    tick_count++;       /* time-slice counter */

    /* Wake any tasks whose delay has expired */
    uint8_t need_switch = task_wake();

    /* Round-robin: has this task used its full slice? */
    if (tick_count >= time_slice_ticks)
    {
        tick_count  = 0U;
        need_switch = 1U;
    }

    if (need_switch)
        request_context_switch();  /* pend PendSV */
}
flowchart LR A["SysTick fires\n(every 1ms)"] --> B["global_systick++\ntick_count++"] B --> C["task_wake()\nCheck sleep list head"] C --> D{{"block_count\n<= systick?"}} D -->|Yes| E["Move to ready queue\nneed_switch = 1"] D -->|No| F{{"tick_count >=\ntime_slice?"}} E --> F F -->|Yes| G["Reset tick_count\nneed_switch = 1"] F -->|No| H{{"need_switch?"}} G --> H H -->|Yes| I["Set PENDSV\n(ICSR register)"] H -->|No| J["Return from SysTick"] I --> J
Figure 8 — SysTick handler decision flow
🧠
Why PendSV instead of switching directly in SysTick?

SysTick has a fixed priority. If we switched context inside it, we'd block any higher-priority interrupts still pending. PendSV has the lowest possible priority (0xFF), so the context switch only happens after every real interrupt handler has finished — giving minimum interrupt latency to your hardware ISRs.

How SVC Reads Its Own Number

The SVC number is encoded in the instruction itself (e.g., SVC #2 is the byte 0x02 two bytes before the return address). The handler retrieves it by reading PC[-2] from the saved stack frame:

SVC_Handler_C — reading the SVC number
static void SVC_Handler_C(uint32_t *stack_frame)
{
    /* stack_frame[6] = saved PC (where SVC was called from) */
    uint8_t *pc         = (uint8_t *)stack_frame[6];
    uint8_t  svc_number = pc[-2];  /* SVC imm8 is 2 bytes back */

    switch (svc_number) { ... }
}
10 — Synchronization

Semaphores — Counting & Signalling

A semaphore is like a ticket counter. Imagine a parking lot with 3 spaces — when all spaces fill, new cars wait. When a car leaves, the next waiting car can enter.

📋

semaphore_t Structure

count — available resources
wq_head — waiting tasks (sorted by priority)
wq_bitmap — priority bitmap of waiters

⬇️

semaphore_wait() — P operation

Decrements count. If count < 0, task is removed from ready queue and placed in semaphore's wait queue.

⬆️

semaphore_post() — V operation

Increments count. If count ≤ 0, the highest-priority waiting task is moved to the ready queue.

Code Example

Producer / Consumer with semaphore
semaphore_t data_ready;

void producer_task(void) {
    semaphore_init(&data_ready, 0); /* start at 0 — consumer must wait */
    while(1) {
        fill_buffer();              /* produce data */
        semaphore_post(&data_ready); /* signal consumer */
        task_delay(100);
    }
}

void consumer_task(void) {
    while(1) {
        semaphore_wait(&data_ready); /* blocks until producer posts */
        process_buffer();
    }
}
11 — Priority Inheritance

Mutex & Priority Inheritance

A mutex is like a toilet key on a keyring. Only one person can have the key at a time. But Kite's mutex is smarter — it prevents a nasty problem called priority inversion.

🚨
The Priority Inversion Problem

A high-priority task waits for a mutex held by a low-priority task. Meanwhile, a medium-priority task preempts the low-priority task — preventing it from ever releasing the mutex. High waits forever! This famously crashed the Mars Pathfinder in 1997.

Kite's Solution: Priority Inheritance

1
LowTask (prio 1)
🔑 holds mutex
HighTask (prio 5)

HighTask wants the mutex — it's locked. HighTask blocks.

2
LowTask (prio → 5)
🔑 still holds mutex
MidTask (prio 3)

propagate_priority() boosts LowTask to prio 5 — it now beats MidTask!

3
LowTask (prio 1)
🔓 released
HighTask (prio 5)

LowTask finishes, releases mutex. Priority restored to 1. HighTask runs immediately.

Priority Restoration on Unlock

svc_mutex_unlock — priority restoration
/* After releasing mutex m, recalculate this task's
   effective priority — it may still hold other mutexes */
if (TCB_EFF_PRIO(current) != TCB_BASE_PRIO(current))
{
    uint8_t max_inherited = TCB_BASE_PRIO(current);

    for (uint8_t i = 0; i < HELD_MUTEX_MAX; i++)
    {
        mutex_t *held = held_get(current, i);
        if (held && held->highest_waiting_prio > max_inherited)
            max_inherited = held->highest_waiting_prio;
    }
    /* Restore to base OR keep boosted if another held
       mutex still has a high-prio waiter */
    update_task_priority(current, max_inherited);
}

Code Example

Protecting shared UART with mutex
mutex_t uart_mutex;

void app_init(void) {
    mutex_init(&uart_mutex);
}

    while(1) {
        mutex_lock(&uart_mutex);   /* acquire exclusive access */
        uart_print("Sensor: 42\r\n");

}
12 — Interrupt Safety

Critical Sections & BASEPRI Masking

A critical section is a region of code that must not be interrupted — like a bank teller who must finish counting money before answering the phone. Kite uses ARM's BASEPRI register to disable only kernel-level interrupts, leaving higher-priority hardware ISRs untouched.

sched_enter_critical / sched_exit_critical
void sched_enter_critical(void)
{
    uint32_t current_basepri;
    __asm volatile(
        "mrs %0, basepri  \n"  /* save current BASEPRI */
        "msr basepri, %1  \n"  /* raise to kernel level (15) */
        "dsb              \n"  /* data sync barrier */
        "isb              \n"  /* instruction sync barrier */
        : "=r"(current_basepri)
        : "r"(KERNEL_INTERRUPT_MASK) : "memory"
    );
    if (critical_nesting == 0U)
        saved_basepri = current_basepri; /* save original on first entry */
    critical_nesting++;                  /* safe to nest */
}

void sched_exit_critical(void)
{
    critical_nesting--;
    if (critical_nesting == 0U)        /* restore only on final exit */
        __asm volatile("msr basepri, %0" :: "r"(saved_basepri) : "memory");
}

✅
Nested critical sections work correctly

If ENTER_CRITICAL() is called twice, only the first call saves the original BASEPRI. Only when EXIT_CRITICAL() brings the nesting counter to 0 is BASEPRI restored. This prevents accidentally re-enabling interrupts mid-nesting.

13 — Hardware Safety

MPU Stack Guard — Catching Stack Overflows

The Memory Protection Unit (MPU) can mark memory regions as no-access. Kite places a 32-byte "guard" region just below each task's stack. If a task overflows its stack into this region, the MPU immediately triggers a MemManage fault — catching the bug before it corrupts other data.

mpu_update_stack_guard — called on every context switch
static void mpu_update_stack_guard(TCB_t *t)
{
    MPU->RNR = STACK_GUARD_MPU_REGION;   /* select region 1 */

    if (t == NULL || t->stack_guard_base == NULL) {
        MPU->RASR = 0U;                 /* disable guard */
        return;
    }

    /* Set base address of this task's guard region */
    MPU->RBAR = ((uint32_t)t->stack_guard_base)
                | MPU_RBAR_VALID_Msk
                | STACK_GUARD_MPU_REGION;

    /* 32 bytes (size=4→2^5), no access (AP=0), no exec (XN) */
    MPU->RASR = MPU_RASR_ENABLE_Msk
              | (4U << MPU_RASR_SIZE_Pos)  /* 2^(4+1) = 32 bytes */
              | (0U << MPU_RASR_AP_Pos)    /* AP=0: no access    */
              | MPU_RASR_XN_Msk;            /* execute never      */
}

Complete Public API

🚀 Kernel Control — kite_start, create_task ▼
FunctionParametersReturnsDescription
kite_start() none never returns Creates the idle task, initializes SysTick/MPU, then fires the first task via SVC. Call after all create_task() calls.
create_task() uint8_t priority, void (*handler)(void), uint32_t stack_words, const char *name 1 success, 0 failure Allocates a TCB from the pool, carves a stack with MPU guard, and adds the task to the ready queue. Priority 0 is lowest (idle), 31 is highest.

set_time_slice_ticks() uint32_t ticks void Changes the round-robin quantum at runtime. Resets tick counter. Minimum 1 tick.
⏱️ Task Timing — task_delay, task_sleep_until, task_yield ▼
FunctionParametersDescription
task_yield() none Triggers SVC #1. Moves the calling task to the back of its priority queue and requests a context switch. Other same-priority tasks get a turn.
task_delay(ticks) uint32_t ticks Blocks the calling task for exactly ticks milliseconds (at default TICK_HZ). Task is inserted into the sleep list, sorted by absolute wake time.
task_sleep_until(*last_wake, period) uint32_t *last_wake, uint32_t period Periodic sleep that corrects for execution drift. Use for precise timing loops — e.g., a control loop that must fire every 10ms regardless of how long the loop body took.
Drift-free periodic task using sleep_until
void control_loop_task(void) {
    uint32_t last_wake = get_systick_counter();
    while(1) {
        run_pid_controller();     /* can take variable time */
        task_sleep_until(&last_wake, 10); /* next wake = last_wake + 10ms */
    }                            /* no drift accumulation! */
}
🚦 Semaphores — semaphore_init, semaphore_wait, semaphore_post ▼
FunctionParametersDescription
semaphore_init(sem, count)semaphore_t *sem, int32_t initial_countInitializes semaphore. Use count=1 for binary, count=N for N resources, count=0 for signalling.
semaphore_wait(sem)semaphore_t *semDecrements count. If count becomes negative, calling task blocks in the semaphore's wait queue (sorted by priority).

🔒 Mutexes — mutex_init, mutex_lock, mutex_unlock ▼
FunctionParametersDescription
mutex_init(m)mutex_t *mInitializes mutex to unlocked state. No owner, no waiters.
mutex_lock(m)mutex_t *mAcquires mutex or blocks. If locked by another task, triggers priority inheritance up the ownership chain.
mutex_unlock(m)mutex_t *mOnly the owner can unlock. Hands ownership to the highest-priority waiter if any. Restores calling task's priority.
⚠️
Mutexes are not binary semaphores

Only the owning task can call mutex_unlock(). Attempting to unlock a mutex you don't own is silently ignored. Unlike semaphores, mutexes implement priority inheritance — always prefer mutex for mutual exclusion, semaphore for signalling.

Build-time Validation

All these constraints are enforced with _Static_assert — if you violate them (e.g., set MAX_TASKS = 32), the compiler will refuse to build with a descriptive error message. No runtime surprises.

Minimal Working Example

main.c — minimal Kite RTOS application
#include "sched.h"

semaphore_t print_sem;
mutex_t     uart_mtx;

void led_task(void) {
    while(1) {
        GPIOC->ODR ^= (1 << 13);
        task_delay(500);
    }
}

void sensor_task(void) {
    uint32_t last = get_systick_counter();
    while(1) {
        int val = read_adc();
        mutex_lock(&uart_mtx);
        uart_printf("ADC=%d\n", val);
        mutex_unlock(&uart_mtx);
        task_sleep_until(&last, 100);
    }
}

int main(void) {
    SystemInit();
    gpio_init();
    uart_init();

    mutex_init(&uart_mtx);
    semaphore_init(&print_sem, 1);

    create_task(3, led_task,    128, "LED");
    create_task(5, sensor_task, 256, "SENSOR");

    kite_start();  /* never returns */
}
